## 2. 与现有项目的关系

现有 v0.3 已经具备计划版本、确认、成交关联、事实卡、研究包和基础纪律事件。新规则不推翻这些数据，而是在其上增加四类不可覆盖对象：

- `disciplineAssessments`：逐笔正式纪律评估；
- `influenceAssessments`：事件针对具体公司的影响评估；
- `probabilityReports`：冻结的概率研报或冷启动证据报告；
- `forecastResolutions`：到期结果和 Brier 分。

旧的 `disciplineEvents` 继续承担即时提示。新评估负责正式打分、完整度、硬规则上限和周期复盘。二者分工如下：

```text
成交录入
  ├─ 即时 disciplineEvents：快速提示是否越界
  └─ disciplineAssessments：补齐记录后形成正式、版本化评分
```

## 3. 接口写入规则

所有规则引擎先允许 `preview`。正式写入必须显式传入 `confirmed: true`，保存：

- 原始输入快照；
- 输入 SHA-256 哈希；
- 规则/报告版本；
- 计算结果；
- 创建时间；
- 审计日志。

不提供覆盖更新接口。修订通过新增记录完成。概率报告结算后禁止再次结算。

## 4. 下一层工程实施顺序

以下是把规则从“后端可调用”接到完整产品界面的顺序，不改变本次规则基线：

1. 在盘中成交记录中补齐风险影响、触发条件、实际/计划风险、情绪和新事实引用；
2. 盘后页面增加逐笔“补齐 → 预览 → 确认评分”流程；
3. 首页增加 20 日纪律画像和唯一首要纠正目标；
4. 建立 `ResearchRun`、`SourceCursor`、`RawDocument`、`EventCluster`，接入合法数据源；
5. 建立行业—产品—公司—客户—供应商—项目关系图；
6. 在标的详情页显示公司传导链和未核验节点；
7. 建立概率报告到期任务、行情结算和校准仪表板；
8. 最后封装 MCP/CLI；HTTP 核心接口保持为共同底座。

## 5. 不允许跳过的验收门

### 5.1 纪律规则

- 盈亏不得进入纪律总分公式；
- 严重风险越界不能被其他维度平均掉；
- 缺关键数据不能自动当作合规；
- 风险降低动作和风险扩大动作必须区别处理；
- 逐笔评分可追溯到当时计划版本；
- 周期分不能通过拆单或增加交易次数刷高；
- 规则更新不改写历史评分。

### 5.2 影响因子

- 原文、转载和独立来源必须区分；
- 来源可靠性和内容可信度必须分别保存；
- 事件影响和公司影响必须分别计算；
- 公司传导链关键节点缺失时不得输出公司影响分；
- 影响分不得显示为概率；
- 所有分析句必须能回到事实引用；
- 更正和撤回不得删除旧事实。

### 5.3 概率研报

- 预测时点、期限、基准和结果阈值必须事前冻结；
- 不达样本量和样本外校准门槛时，不显示概率；
- 到期必须自动或人工确认结算；
- 结算口径不能事后修改；
- 显示 Brier 分、基线和样本量，而不是只显示命中率；
- 研报只产生条件化观察与计划调整建议，不自动下单。

## 6. 测试基线

`test/decision-engines.test.js` 当前覆盖：

1. 合规盈利和合规亏损同分；
2. 无计划新增风险的硬上限；
3. 关键记录不足不出正式分；
4. 紧急纯减仓不误罚；
5. 周期汇总惩罚最差决策并识别重复违规；
6. 公司传导链缺口阻止公司影响分；
7. 明确反证使传导链断裂；
8. 冷启动隐藏伪概率，合格模型可结算 Brier 分。

API 闭环已加入集成测试。发布前仍需补充专门的旧数据迁移测试、并发写入测试、前端交互测试和实际数据源契约测试。
