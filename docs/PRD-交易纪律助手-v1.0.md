# 交易纪律助手（Trading Discipline Assistant）产品需求文档

> PRD 版本：v1.0<br>
> 目标产品阶段：v0.3–v1.0<br>
> 文档状态：需求基线 / 待用户确认<br>
> 更新日期：2026-07-18<br>
> 部署方式：本地优先、单用户、默认离线<br>
> 保密级别：个人私密；本文不包含真实持仓、金额、成交或个人规则内容

## 0. 执行摘要

交易纪律助手是一款面向个人投资者的本地部署交易辅助工具。它不预测涨跌、不代替用户下单，也不以单笔盈亏判断决策好坏；它的核心价值是把“事前计划—盘中记录与偏差识别—盘后复盘—次日计划”固化为可量化、可追溯、可持续改进的个人交易系统。

产品最重要的判断标准是：**纪律优先于单笔盈亏，过程质量与结果盈亏分开评价。** 违规盈利仍记为纪律问题；合规亏损不自动判定为错误。任何计划、计划调整和交易动作都要记录依据、版本、确认人、确认时间、执行结果和信息来源。

首要建设重点不是增加更多行情图，而是补齐当前产品最关键的结构性缺口：

- 将“当前持仓”和“计划交易标的”彻底分开；
- 建立可确认、可版本化、可失效的交易计划；
- 建立事实、分析、用户判断、系统建议四层信息结构；
- 建立盘中偏差识别和风险扩大识别；
- 建立纪律评分、证据链和长期改进指标；
- 在不泄露账户数据的前提下，提供本地定时信息整理和风险提示。

## 1. 文档目的与范围

### 1.1 文档目的

本 PRD 用于统一产品方向、功能边界、数据结构、判断规则、隐私要求和版本路线，作为 v0.3 以后产品设计、开发、测试和验收的需求基线。

### 1.2 本期范围

- A 股沪深市场的个人持仓、候选标的、交易计划、交易记录与复盘；
- 交易日历与盘前、午间、盘后工作流；
- 公开信息采集、来源标注、事实整理与情景分析；
- 本地规则引擎、纪律评分、风险提醒、版本和审计记录；
- 数据导入、备份、恢复、脱敏导出与本地安全。

### 1.3 明确不做

- 不承诺预测集合竞价、次日涨跌、最高点、最低点或“雪崩”；
- 不向用户给出确定性买卖结论，不替用户做最终决策；
- 不自动登录券商、自动下单、自动撤单或操作用户资金；
- 不以收益率作为唯一或首要产品成功指标；
- 不使用无法说明来源、时间或完整性的“黑箱数据”；
- 未经用户确认，不让 AI 或程序静默修改计划、规则、持仓或账本。

## 2. 产品定位

### 2.1 一句话定位

面向个人投资者的本地部署交易纪律与风险管理助手。

### 2.2 核心用户

主要用户为产品所有者本人；用户具备自主交易判断，会在同花顺等券商或行情终端完成交易，希望通过工具减少冲动交易、规范计划修改、保留完整决策证据并逐步形成稳定的个人交易系统。

### 2.3 用户核心任务（JTBD）

- 交易前：把模糊想法转化为有触发条件、仓位上限、风险上限和失效条件的计划；
- 盘中：交易后快速记录，让系统判断是否偏离已确认计划、是否冲动、是否扩大风险；
- 盘后：客观还原当日事实、计划、调整、执行和结果，不用盈亏替代纪律判断；
- 次日前：结合持仓风险、公开信息和用户判断，形成待确认的情景计划；
- 长期：用指标发现反复出现的纪律问题，修改个人规则并验证改进效果。

## 3. 产品目标与成功标准

### 3.1 产品目标

1. 降低无计划交易和临时冲动交易的频率。
2. 让每个交易动作都能关联到已确认计划或有依据的计划调整。
3. 将事实、分析、判断和结果分开保存，避免事后合理化。
4. 规范交易记录、资金明细和个人规则，形成一致的数据结构。
5. 建立可以按周、月、季度复盘的纪律指标和规则迭代机制。

### 3.2 北极星指标

**已确认计划覆盖率**：有交易行为的标的中，交易前存在有效且已确认计划的比例。

### 3.3 核心过程指标

| 指标 | 定义 | 目标方向 |
|---|---|---|
| 无计划交易率 | 无有效确认计划的成交笔数 / 总成交笔数 | 持续下降 |
| 计划外风险扩大率 | 实际风险超过确认计划上限的交易笔数 / 总成交笔数 | 持续下降 |
| 计划调整证据覆盖率 | 有来源、有理由、有确认时间的调整数 / 总调整数 | 接近 100% |
| 盘后复盘完成率 | 已完成复盘的交易日 / 应复盘交易日 | 持续上升 |
| 违规盈利识别率 | 盈利但存在纪律违规且被正确标记的事件比例 | 接近 100% |
| 合规亏损误判率 | 合规亏损被错误判为纪律问题的比例 | 接近 0 |
| 数据健康度 | 数据源可用、时效符合、关键字段完整的综合比例 | 稳定可见 |
| 恢复验证通过率 | 定期备份可成功恢复的次数 / 恢复演练次数 | 100% |

收益、胜率、回撤可作为交易结果指标展示，但不能替代纪律指标，也不作为系统“建议准确率”。

## 4. 核心原则

### 4.1 纪律优先于单笔盈亏

- 违规盈利：结果为盈利，但计划状态、风险上限、禁止事项或确认流程被破坏，纪律结论仍为不合规；
- 合规亏损：交易符合有效计划和风险边界，亏损本身不构成纪律错误；
- 结果评价和过程评价必须使用不同字段、不同分数、不同结论。

### 4.2 计划先于交易

- 默认要求交易前存在已确认且未失效的计划；
- 突发利好、利空或价格异动只能触发“重新评估”，不能自动成为追涨杀跌理由；
- 临时灵活调整允许存在，但必须有明确幅度、风险上限、事实依据和确认机制；
- 信息不足时执行预先设置的默认动作，通常是“不新增风险、等待确认”。

### 4.3 事实判断与分析分离

每条信息必须归入至少一种类型：

- 事实：行情、公告、财报、政策原文、成交记录等可验证信息；
- 分析：对事实可能影响方向、持续时间、传导路径的解释；
- 用户判断：用户对市场、标的和风险的主观看法；
- 系统建议：基于规则和情景生成的待确认建议。

“公告内容偏利好”属于分析，不等于“当日价格一定上涨”。系统不能把二者合并成确定性结论。

### 4.4 用情景分析替代确定性预测

至少保留基准、乐观、悲观三种情景。每个情景包含触发条件、观察指标、允许动作、禁止动作、最大风险和失效条件。情景概率可选填，但不得显示为确定事实。

### 4.5 不承诺准确预测

系统在界面、报告和通知中禁止使用“必涨、必跌、抄底、逃顶、确定性雪崩”等表达。对集合竞价、次日方向和极端事件仅做数据整理、风险枚举和条件化推演。

### 4.6 确认与追溯

- 草稿不能直接成为执行计划；
- 每个版本记录修改前后差异、修改原因、依据、创建时间、确认时间和确认方式；
- 用户确认后生成不可覆盖的版本快照；
- 后续调整通过新版本完成，不直接改写历史；
- 交易、复盘和纪律事件必须能回溯到当时有效的具体计划版本。

### 4.7 本地优先与最小暴露

- 真实持仓成本、交易记录、资金明细、个人规则默认仅保存在本地；
- 默认不上传云端、不发送遥测、不进入 Git；
- 调用外部模型或服务前，先生成可预览的脱敏信息包并由用户确认；
- 第三方密钥存放在本机安全凭据区，不写入源码、日志或导出文件。

## 5. 核心概念与状态

### 5.1 关键概念

| 概念 | 定义 |
|---|---|
| 当前持仓 | 已实际持有的证券及成本、数量、可用数量、资金占用等事实 |
| 计划交易标的 | 用户正在研究或准备交易的证券，不要求已持仓 |
| 交易计划 | 面向一个交易日或一段有效期的条件化执行规则 |
| 计划版本 | 一次完整、不可覆盖的计划快照 |
| 计划调整 | 计划生效后因新事实或风险变化产生的新版本 |
| 纪律事件 | 无计划交易、追涨、风险扩大、违反禁忌等可量化行为 |
| 证据 | 支持事实或分析的来源、发布时间、抓取时间和原文定位 |

### 5.2 计划状态机

计划状态依次为：草稿 → 待确认 → 已确认 → 生效中 → 已调整 / 已失效 / 已完成。

- 草稿：可编辑，不可执行；
- 待确认：内容冻结，等待用户确认或退回；
- 已确认：具有确认记录，但尚未到有效时间；
- 生效中：允许关联交易；
- 已调整：原版本保留，新版本接替；
- 已失效：触发失效条件或用户主动终止；
- 已完成：有效期结束且已进入复盘。

## 6. A 股交易日闭环

### 6.1 时间轴与系统任务

| 时间 | 市场阶段 | 系统任务 | 用户动作 | 产物 |
|---|---|---|---|---|
| T-1 收盘后 | 盘后 | 对账、还原操作、纪律复盘、公开信息整理 | 补充当日判断和复盘说明 | 当日复盘、次日计划草稿 |
| 08:00 | 盘前 | 扫描持仓与计划标的的公告、政策、国际事件、上下游与隔夜风险 | 查看变化，补充判断 | 盘前风险简报、调整建议 |
| 09:15–09:25 | 开盘集合竞价 | 读取授权行情或接收用户手工观察；只做状态和风险提示 | 输入观察或确认数据 | 集合竞价观察卡 |
| 09:25–09:30 | 衔接 | 冻结盘前事实快照，提示未确认或已失效计划 | 最终确认 / 暂停 | 上午有效计划版本 |
| 09:30–11:30 | 上午连续竞价 | 接收交易记录，实时比对计划和风险边界 | 成交后快速记录 | 执行记录、偏差提示 |
| 11:30–13:00 | 午间休市 | 整理上午行情、公告、事件和计划偏差 | 确认下午调整 | 午间风险简报、下午计划版本 |
| 13:00–14:57 | 下午连续竞价 | 持续比对计划，提示风险扩大和临近失效 | 成交后记录 | 执行记录、风险事件 |
| 14:57–15:00 | 收盘集合竞价 | 标记收盘集合竞价阶段；应用相应价格和撤单规则提示 | 根据计划执行或不行动 | 收盘阶段记录 |
| 15:05–15:30 | 盘后固定价格交易 | 对符合规则的 A 股、ETF 展示盘后固定价格阶段和风险提示 | 通过券商端自行申报并记录 | 盘后固定价格交易记录 |
| 15:30 后 | 盘后 | 对账、事实锁定、纪律评分、次日信息清单 | 完成复盘并给出次日判断 | 正式复盘、次日计划草稿 |

注：以上交易时段按 2026-07-06 起生效的沪深交易规则整理。交易日由交易所休市安排驱动，不按普通工作日简单推算。

### 6.2 盘前 08:00 简报

系统应按“当前持仓 + 计划交易标的”生成简报，覆盖：

- 上市公司公告、监管问询、业绩和重大事项；
- 国家政策、行业政策和监管变化；
- 上下游公司、关键供应商、客户、竞争对手和替代品事件；
- 国际形势、海外市场、商品、汇率、利率与相关产业风险；
- T-1 收盘后至当日 08:00 的新增信息；
- 对既有计划的影响检查：无影响 / 需关注 / 需调整 / 计划失效；
- 信息缺口和待用户补充项。

系统输出只能是待确认调整建议。用户未确认时，原计划保持不变；若原计划触发硬失效条件，则状态改为“暂停执行”，但不能自动创建新交易动作。

### 6.3 集合竞价双方案

**方案 A：授权行情数据模式。** 当用户配置合法、可用且满足实时频率的行情产品时，系统可读取虚拟参考价格、虚拟匹配量、未匹配量、买卖档位和行情时间戳，展示变化过程与计划条件是否被触发。系统不把竞价强弱直接翻译成必涨必跌。

**方案 B：用户手工观察模式。** 当没有授权接口时，提供结构化输入：价格相对昨收、虚拟匹配量变化、未匹配方向、是否接近涨跌停、撤单变化、用户观察描述。系统只做完整性检查、与计划的条件匹配和风险提示，不声称独立识别“抢筹”或“砸盘”。

默认采用方案 B。方案 A 必须通过数据授权、字段完整性、时间延迟和稳定性验收后才能启用。

### 6.4 午间重评

午间任务只处理 09:25 后新增事实和上午执行情况：

- 上午已发生的交易、风险暴露和偏差；
- 上午公告、监管信息、新闻和行业事件；
- 指数、成交额、市场宽度、行业强弱和量价结构；
- 计划触发条件是否成立、是否需要降低风险或失效；
- 下午允许动作、禁止动作和待确认项。

## 7. 信息架构与主要页面

### 7.1 首页：纪律驾驶舱

- 今日阶段、下一个时间节点和未完成任务；
- 当前总风险、单标的风险、可用风险预算；
- 有效计划、待确认调整、已失效计划；
- 盘中风险提示与数据源健康状态；
- 快速记录交易、判断、情绪和计划调整；
- 今日纪律状态，不以当日盈亏作为主视觉。

### 7.2 持仓

- 当前数量、可用数量、成本、现价、市值、浮动盈亏；
- 资金和持仓流水，支持对账与差异处理；
- 隔夜风险、关键事件和关联计划；
- 真实数据默认不进入任何公开导出。

### 7.3 计划交易标的

这是独立于持仓的一级模块。每个标的包含：

- 预期收益：金额或百分比，可选；必须同时展示最大可接受风险和风险收益比；
- 用户市场判断、系统市场整理，二者分栏显示；
- 基本面、估值、盈利变化；
- 技术面、K 线、长期趋势；
- 量价结构、成交额、相对强弱；
- 减仓条件、止损退出条件、最大仓位、单笔最大风险；
- 当日禁忌 / 禁止事项；
- 计划失效条件；
- 信息不足时的默认动作；
- 持仓和公开信息的隔夜风险；
- 次日关键价格和关键事件；
- 基准、乐观、悲观三种情景；
- 下一交易日暂定计划；
- 需要获取的信息清单。

### 7.4 计划中心

- 今日计划、次日草稿、历史版本；
- 计划确认、调整对比、失效原因；
- 规则触发记录和关联交易；
- 用户可一键查看“当时我知道什么、当时为何修改、当时确认了什么”。

### 7.5 盘中记录

- 快速输入买入、卖出、撤单、未成交、观察、情绪；
- 成交可先快速记录，盘后再与券商账单核对；
- 输入后立即显示：关联计划、触发条件、风险变化、纪律提示；
- 不阻止用户保存“违规动作”，而是准确记录并提示。

### 7.6 复盘与改进

- 时间线还原：事实 → 判断 → 计划 → 调整 → 交易 → 结果；
- 过程评分和结果统计分开；
- 违规盈利、合规亏损、合规盈利、违规亏损四象限；
- 周/月纪律趋势、反复违规模式、规则变更效果；
- 生成待确认的个人规则改进建议。

### 7.7 信息与证据库

- 事实卡、分析卡、来源、抓取时间、发布时间和原文链接；
- 持仓、标的、行业、上下游和宏观主题之间的关联；
- 用户可标记可信度、相关性、是否已读、是否已用于计划；
- 旧事实不能被新内容覆盖，只能追加更正或失效标记。

## 8. 功能需求

### FR-01 本地身份与设置（P0）

- 单用户本地配置，支持本机登录密码或操作系统账户保护；
- 时区固定为 Asia/Shanghai，交易日历按沪深交易所公告更新；
- 支持风险单位、默认仓位上限、复盘时间和通知方式设置；
- 首次启动必须完成隐私和免责声明确认。

### FR-02 持仓与资金账本（P0）

- 支持持仓、资金余额、出入金、交易费用、股息、税费和其他资金事件；
- 所有写入使用追加式流水，错误通过冲正或更正记录处理；
- 支持从券商导出文件导入，并先进入预览与差异确认；
- 资金合计、持仓数量和交易记录之间应可对账。

### FR-03 计划交易标的（P0）

- 支持无持仓标的进入观察和计划；
- 支持状态：观察中、研究中、待计划、已有计划、暂停、归档；
- 标的档案可关联多个计划，但同一交易时段只允许一个主执行版本；
- 支持上下游、行业、指数、商品、汇率和关键事件关联。

### FR-04 交易计划编辑器（P0）

- 计划必须包含：方向、有效期、情景、触发条件、允许动作、仓位上限、单笔风险、退出条件、禁止事项、失效条件和默认动作；
- 关键数值支持金额、价格、百分比和仓位比例；
- 缺少 P0 字段时只能保存草稿，不能提交确认；
- 预期收益是目标字段，不是系统承诺，不能替代退出和风险条件。

### FR-05 确认、版本与追溯（P0）

- 草稿提交后生成待确认快照；
- 用户必须明确点击确认，并可填写确认说明；
- 计划生效后的修改自动创建新版本和差异摘要；
- 每次确认记录本地时间、计划哈希、确认方式和确认人；
- 历史版本只读，任何导出都包含版本号。

### FR-06 公开信息采集（P1）

- 优先顺序：交易所 / 巨潮等法定披露 → 政府与监管机构 → 公司官方 → 权威行业与市场数据 → 其他来源；
- 每条信息保存标题、来源、原始链接、发布时间、抓取时间、覆盖标的和信息类型；
- 重复内容去重，但保留多个独立来源；
- 抓取失败、延迟、字段缺失必须在数据健康页显示；
- 不允许用搜索摘要代替公告原文事实。

### FR-07 事实与分析分层（P0）

- 事实卡必须有来源和时间；
- 分析卡必须引用至少一个事实卡，或明确标记“基于用户判断”；
- 系统生成内容显示生成时间、输入范围和不确定性；
- 用户可以修订分析，但不能改写原始事实；
- 事实更正通过新版本和更正关系表达。

### FR-08 市场环境整理（P1）

系统不做涨跌预测，只整理并展示：

- 指数趋势、全市场成交额、量价结构；
- 上涨 / 下跌家数、涨跌加速、市场宽度；
- 行业强弱、相对强度、涨跌停与情绪指标；
- 公开可得的资金指标及其披露频率；
- 日 K、周 K、月 K 和长期趋势；
- 公司基本面、估值和盈利变化；
- 政策、行业、外部市场、商品、汇率和利率影响。

北向资金不再假设存在实时买卖成交额。系统按公开披露频率展示收市后成交总额、活跃证券或其他仍可得指标，并明确“实时不可得 / 延迟 / 低频”。

### FR-09 盘前与午间研究任务（P1）

- 08:00 任务和 11:30 任务分别独立运行；
- 任务具备幂等性，同一时间窗重复运行不会生成重复事实；
- 新信息与 T-1 计划自动做影响匹配；
- 只生成“维持 / 关注 / 调整 / 失效”建议，不自动买卖；
- 调整必须进入确认流程。

### FR-10 盘中交易记录（P0）

- 交易动作字段：时间、证券、方向、价格、数量、订单状态、渠道、备注；
- 决策字段：当时计划版本、触发条件、用户理由、情绪、紧迫感；
- 支持先录入后对账，保留原始录入时间和最终成交时间；
- 支持语音或快捷模板作为未来增强，但最终需结构化确认。

### FR-11 计划偏差与风险扩大识别（P0）

系统在记录后检查：

- 是否存在有效确认计划；
- 动作是否属于允许动作；
- 触发条件是否已满足；
- 实际仓位是否超过最大仓位；
- 单笔和组合风险是否超过上限；
- 是否违反当日禁忌；
- 是否在计划已失效后继续交易；
- 是否以“新信息”为理由但没有来源或确认；
- 是否出现追涨、报复性交易、连续加码、亏损后扩大风险等模式。

识别结果分为提示、警告、严重违规；系统说明触发的规则和证据，不只给标签。

### FR-12 灵活调整边界（P0）

- 每个计划可定义“允许微调范围”，如价格容差、数量容差和时间容差；
- 容差内调整仍需记录理由，可采用快速确认；
- 超出容差、扩大仓位、放宽止损或新增风险属于重大调整，必须完整确认；
- 重大调整在确认前默认不建议执行；
- 紧急减仓或退出可先执行后补充说明，但必须保留“紧急风险处置”标记。

### FR-13 风险提示（P0）

- 硬提示：无有效计划、计划失效、仓位超限、单笔风险超限、数据严重过期；
- 软提示：接近关键价格、事件临近、信息不足、情绪异常、连续交易频率异常；
- 提示必须包含原因、当前值、计划值、建议默认动作和确认入口；
- 提示不应使用煽动性文案，不通过重复弹窗诱发交易。

### FR-14 盘后复盘（P0）

- 自动生成当日事实时间线和待补充项；
- 逐笔比较计划、实际动作、风险和结果；
- 用户补充当时判断、情绪、遗漏信息和改进点；
- 输出纪律结论、严重程度、证据、是否重复发生；
- 复盘完成后锁定版本，如需修改生成修订版。

### FR-15 纪律评分（P0）

建议总分 100，组成如下：

- 计划完整度 20；
- 交易前确认 15；
- 触发条件遵守 20；
- 仓位与风险遵守 20；
- 调整依据与确认 15；
- 记录和复盘完整度 10。

盈亏不进入纪律总分。结果指标单独显示。评分规则可配置、可版本化；规则变更后不追溯改写历史分数，除非用户主动执行重算并保留旧分数。

### FR-16 次日计划生成（P0）

- 输入：当日复盘、收盘持仓、有效个人规则、公开事实、用户判断、风险预算；
- 输出：关键价格、关键事件、隔夜风险、三种情景、暂定动作、禁止事项、失效条件、信息清单；
- 输出始终为草稿，必须由用户确认；
- 缺少核心数据时标记不完整，并采用默认动作，不编造结论。

### FR-17 周期性改进（P1）

- 周报：无计划交易、风险扩大、重复违规、计划覆盖和复盘完成；
- 月报：纪律趋势、规则有效性、情绪模式、交易费用与换手影响；
- 规则建议：说明证据、影响样本和预期改善，进入待确认状态；
- 支持对规则版本前后进行对比，但不将小样本变化解释为确定因果。

### FR-18 数据导入、备份与脱敏导出（P0）

- 导入必须先预览、校验、去重和确认；
- 自动本地备份，支持用户指定加密备份位置；
- 定期恢复演练并记录结果；
- 导出时可选择隐藏证券、价格、数量、金额、备注和规则正文；
- 用于 Git 的导出仅包含程序、结构和示例数据，不包含真实数据。

### FR-19 本地信息脚本与通知（P1）

- 本地调度器执行 08:00、11:30、15:30 后任务；
- 支持 Windows 本地通知、应用内通知和可选邮件 / 飞书等外部通知；
- 外部通知默认只发送“有待确认风险”，不发送持仓、成本和金额；
- 每次任务有运行日志、数据源状态、重试和失败说明。

### FR-20 系统边界与免责声明（P0）

- 首屏和关键建议处明确“工具用于纪律和信息整理，不构成投资建议”；
- 用户始终负责最终判断和券商端操作；
- 系统不能用模拟分析冒充实时行情；
- 所有建议必须可解释、可回溯、可拒绝。

## 9. 纪律规则引擎

### 9.1 判定维度

规则引擎按以下顺序判断，避免被盈亏结果干扰：

1. 当时是否存在有效、已确认的计划；
2. 交易是否满足计划触发条件；
3. 动作、数量和价格是否在允许范围；
4. 是否违反仓位、单笔风险或组合风险上限；
5. 调整是否有新事实、理由和确认；
6. 是否违反禁忌或计划失效条件；
7. 记录是否完整；
8. 最后才单独记录盈亏结果。

### 9.2 过程与结果四象限

| 纪律 | 结果 | 结论 | 复盘重点 |
|---|---|---|---|
| 合规 | 盈利 | 合规盈利 | 是否可重复、是否按计划兑现 |
| 合规 | 亏损 | 合规亏损 | 情景是否覆盖、风险是否可接受，不贴“错误”标签 |
| 违规 | 盈利 | 违规盈利 | 防止结果强化坏习惯，明确违反的规则 |
| 违规 | 亏损 | 违规亏损 | 纪律问题优先，结果只说明代价 |

### 9.3 严重程度

- 提示：轻微偏差，未扩大风险，处于预设容差内；
- 警告：超出容差、缺少确认或理由不足，但未突破硬风险上限；
- 严重违规：无计划新增风险、突破仓位 / 风险上限、计划失效后交易、放宽止损后继续加码；
- 紧急风险处置：未经完整流程但以减小风险为唯一目的，单独记录，不等同于普通违规。

### 9.4 默认动作

当出现信息不足、数据源故障、观点冲突或计划不完整时，默认动作必须在个人规则中预设。系统默认建议为：

- 不新增风险；
- 不提高仓位上限；
- 不放宽退出条件；
- 等待用户确认或可靠信息补齐；
- 若已触发硬退出条件，则优先执行既有风险处置计划。

## 10. 数据结构

### 10.1 核心实体

| 实体 | 关键字段 | 说明 |
|---|---|---|
| Security | 市场、代码、名称、证券类别、交易规则 | 证券主数据 |
| Position | 数量、可用数量、成本、市值、更新时间 | 当前持仓快照 |
| CashLedger | 时间、类型、金额、来源、关联交易 | 资金追加式流水 |
| PlannedTarget | 证券、研究状态、用户判断、关联主题 | 计划交易标的 |
| PersonalRule | 规则类型、阈值、正文、版本、有效期 | 个人规则版本 |
| MarketFact | 类型、内容、发布时间、抓取时间、来源 | 不可覆盖事实卡 |
| AnalysisNote | 观点、依据事实、作者、置信说明 | 分析层 |
| Scenario | 基准 / 乐观 / 悲观、触发、动作、失效 | 条件化情景 |
| TradePlan | 标的、交易日、状态、主版本 | 计划容器 |
| PlanVersion | 规则快照、差异、理由、哈希 | 不可覆盖版本 |
| Confirmation | 对象、时间、方式、确认说明 | 用户确认记录 |
| TradeRecord | 时间、方向、价格、数量、状态、来源 | 交易记录 |
| DisciplineEvent | 规则、严重度、证据、处理状态 | 纪律事件 |
| Review | 交易日、时间线、评分、结论、版本 | 盘后复盘 |
| ResearchRun | 时间窗、数据源、状态、错误、产物 | 信息任务运行记录 |
| RiskAlert | 类型、阈值、当前值、状态、确认 | 风险提示 |

### 10.2 关联约束

- TradeRecord 必须关联 PlanVersion，或明确标记“无计划”；
- AnalysisNote 必须关联 MarketFact 或明确标记“纯用户判断”；
- PlanVersion 必须关联至少一个 Scenario；
- Confirmation 必须保存对象哈希，防止确认后内容被静默替换；
- Review 必须保存当日使用的个人规则版本和计划版本；
- 删除使用软删除或归档，审计对象不得物理删除。

## 11. 数据与信息处理原则

### 11.1 来源等级

1. L1：交易所、证监会、政府部门、法定信息披露平台；
2. L2：上市公司官网、官方投资者关系渠道、行业主管机构；
3. L3：授权市场数据服务商、权威媒体和研究机构；
4. L4：其他媒体、社交平台和用户转述。

系统同时展示来源等级和原始链接。低等级来源可以作为线索，不能在未交叉验证时成为硬规则调整的唯一依据。

### 11.2 数据卡标准字段

- 事实 / 分析类型；
- 标的和主题；
- 发布时间、抓取时间、行情时间；
- 来源等级和原始链接；
- 数据频率、延迟和覆盖范围；
- 完整 / 缺失 / 过期 / 冲突状态；
- 对计划的关联和影响说明；
- 用户是否已读、是否已确认使用。

### 11.3 数据新鲜度

- 实时行情：以供应商时间戳和延迟等级为准；
- 公告：以交易所 / 法定披露发布时间为准；
- 财务和估值：显示报告期和计算日期；
- 政策：显示发布机构、生效日期和原文；
- 新闻：显示原始事件时间和报道时间，不能只保存抓取时间。

## 12. 隐私、安全与保密

### 12.1 数据分类

- 最高敏感：持仓成本、交易记录、资金明细、个人规则、账户标识；
- 敏感：用户判断、复盘文字、情绪记录、候选标的；
- 一般：公开行情、公告、政策、公开财务数据；
- 可公开：程序源码、空数据库结构、虚构示例和无个人信息文档。

### 12.2 安全要求

- 数据库和备份支持静态加密；
- 本地文件采用最小权限，密钥不与数据库同目录明文存放；
- 日志默认不记录持仓、成本、金额、完整规则正文和外部密钥；
- Git 忽略 data、reports、logs、备份、密钥和私有脚本；
- 外部 AI 调用默认关闭，启用时需预览和确认脱敏内容；
- 支持一键生成隐私检查报告，列出可能外泄的文件和字段；
- 支持退出时自动锁定，以及空闲锁定；
- 更新程序不能覆盖本地数据，升级前自动生成可恢复备份。

### 12.3 AI 数据边界

AI 可以读取用户明确授权的脱敏上下文，生成计划草稿、影响分析和复盘摘要；AI 不得直接写入资金账本、成交事实或已确认计划。任何 AI 结果进入正式记录前必须由用户确认。

## 13. 非功能需求

| 类别 | 要求 |
|---|---|
| 可用性 | 核心账本、计划、记录和复盘在无网络时可用 |
| 可靠性 | 写入具备事务性；定时任务幂等；失败不产生半条记录 |
| 性能 | 本地核心页面 2 秒内打开；交易快速记录 3 步内完成 |
| 可解释性 | 每个警告显示触发规则、计划值、实际值和证据 |
| 可追溯性 | 计划、规则、复盘、确认和导入均有版本及审计 |
| 兼容性 | 优先支持 Windows 本地运行；浏览器界面适配桌面与窄屏 |
| 可恢复性 | 自动备份；支持恢复预览；恢复不覆盖现有库前再次确认 |
| 可观察性 | 数据源、调度器、备份、导入和规则引擎均有健康状态 |
| 可维护性 | 数据源适配器与规则引擎解耦，交易规则按市场和日期版本化 |

## 14. 关键交互与文案原则

### 14.1 确认交互

确认页必须同时显示：修改内容、修改原因、使用的事实、风险变化、失效条件和确认后的有效时间。按钮使用“确认此版本”“退回修改”“暂停执行”，避免模糊的“确定”。

### 14.2 风险提示交互

提示应简洁、非煽动、可行动。例如：

> 计划外风险扩大：本次买入后仓位将由 28% 升至 36%，超过计划上限 30%。当前计划不允许该动作。默认动作：不新增风险；如需调整，请先创建并确认新版本。

### 14.3 复盘交互

先呈现客观时间线，再让用户填写解释；先判纪律，后展示盈亏。系统不得用“今天赚钱，所以操作正确”或“今天亏钱，所以判断错误”的结论性文案。

## 15. 优先级与版本路线

### v0.3：计划与证据基座

- 计划交易标的；
- 计划编辑器、状态机、版本、确认和失效；
- 事实 / 分析 / 用户判断分层；
- 交易记录关联计划版本；
- 基础偏差识别；
- 隐私检查、备份和恢复预览。

### v0.4：交易日闭环

- 08:00 盘前任务、11:30 午间任务、盘后任务；
- A 股交易日历和市场阶段；

## 16. MVP 验收标准

### AC-01 无计划交易

当用户记录一笔买入且不存在生效中的已确认计划时，系统保存该笔记录，并立即标记“无计划新增风险”的严重纪律事件；系统不能拒绝保存或伪造关联计划。

### AC-02 违规盈利

当一笔交易违反最大仓位规则但最终盈利时，复盘结果必须同时显示“结果：盈利”和“纪律：严重违规”，纪律评分不因盈利而自动修复。

### AC-03 合规亏损

当交易完全符合确认计划且在预设止损条件退出，即使亏损，系统应标记“合规亏损”，并将问题聚焦于情景覆盖和规则适用性，而不是直接判错。

### AC-04 计划调整

当用户将最大仓位从 30% 调整为 40% 时，系统必须创建新版本，要求填写依据，显示风险变化，并在确认前保持旧版本有效或将执行状态设为暂停；不能静默覆盖 30%。

### AC-05 新公告影响

08:00 任务发现持仓公司新增重大公告时，系统建立事实卡并关联原文，生成“需关注 / 调整 / 失效”建议。未经用户确认，不自动改变交易动作。

### AC-06 信息不足

当关键行情或公告数据源不可用时，系统显示缺失和过期状态，执行预设默认动作，不得补造数据或给出确定性判断。

### AC-07 隐私导出

选择“用于 Git 的安全导出”后，结果中不得出现真实证券、持仓、成本、资金、交易、个人规则正文、外部密钥和日志；安全检查必须返回可核验清单。

### AC-08 交易日阶段

系统在交易日正确识别 09:15–09:25 开盘集合竞价、09:30–11:30 与 13:00–14:57 连续竞价、14:57–15:00 收盘集合竞价、15:05–15:30 盘后固定价格交易，并在休市日不运行交易日任务。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 把分析误当事实 | 形成错误计划或事后合理化 | 强制分层、事实必须有来源、分析引用事实 |
| 免费接口不稳定或无授权 | 盘中提示错误、合规风险 | 默认手工模式；授权后才启用实时接口；显示数据健康 |
| 提示过多引发焦虑或频繁交易 | 工具反而放大冲动 | 分级、合并、静默窗口；默认动作偏向不新增风险 |
| 用户事后修改记录 | 破坏追溯性 | 追加式修订、版本和时间戳，不覆盖原始记录 |
| AI 过度自信 | 用户误把建议当预测 | 禁用确定性文案、展示输入与不确定性、用户确认 |
| 本地设备故障 | 数据丢失 | 加密备份、恢复演练、升级前快照 |
| 个人规则越来越复杂 | 难以执行 | 限制规则层级、提供冲突检测和规则精简建议 |

## 18. 待用户确认的产品决策

1. 预期收益默认采用“金额、百分比或二者都可”，是否需要强制同时填写最大风险？建议：必须同时填写风险。
2. 微调容差如何设置：按价格、数量、仓位还是时间分别配置？建议：四类独立配置。
3. “紧急减仓先执行后补确认”的有效时间窗是多少？建议：当日复盘前补齐。
4. 纪律评分是否采用建议的 100 分结构，还是完全按个人规则自定义？建议：系统模板 + 用户可版本化修改。
5. 是否购买或已有合法行情数据源，以启用集合竞价方案 A？默认：无，先做方案 B。
6. 外部信息服务和 AI 是否允许接收脱敏后的证券代码或行业信息？默认：逐次预览确认。
7. 外部通知渠道是否启用？默认：仅本机通知，外部通知隐藏持仓和金额。
8. 本地数据库密码和备份密钥遗失后的恢复策略如何处理？建议：明确“无法找回”并提供恢复密钥离线保管流程。

## 20. 结论

本产品的最佳发展方向不是“更会猜行情”，而是成为一套可信的个人交易操作系统：在交易前把判断变成规则，在交易中让偏差及时可见，在交易后把过程与结果分开评价，并通过长期指标推动规则迭代。

v0.3 应首先完成“计划交易标的 + 计划版本确认 + 事实分析分层 + 交易关联 + 纪律判断”五项基础能力。只有这些基础稳定后，再增加盘前研究、午间更新、授权实时行情和更多通知，才不会把信息量变成新的冲动来源。
