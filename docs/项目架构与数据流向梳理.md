# 交易纪律助手 - 项目架构与数据流向梳理

> **核心目标**：保护本金、减少临时决策、让账户数字可追溯
> **技术栈**：Vue 3 + Vite + Node.js + SQLite
> **数据原则**：事件溯源（只追加不覆盖）、本地私密、AI 辅助但不替代决策

### 3.2 各阶段数据流转详解

#### 阶段一：研究准备（标的中心 → 证据 → 传导关系）

```
数据输入：
  · 人工添加候选标的 (plannedAssets)
  · 导入研究包 (researchSnapshots → 拆分为 plannedAssets + evidenceRecords + companyRelations)
  · 资讯采集 (informationEvents → AI 整理后可生成影响评估)

数据加工：
  1. 标的池管理：plannedAssets (观察中 → 研究中 → 待计划 → 已有计划 → 暂停)
  2. 证据卡分层：evidenceRecords
     - fact（事实）：可验证的客观信息
     - analysis（分析）：基于事实的推理
     - user_judgment（用户判断）：主观观点（与事实分开存储）
  3. 传导关系建模：companyRelations
     - 9个阶段：需求产生 → 预算资金 → 产品匹配 → 采购路径 → 公司资格
               → 历史准入 → 产能交付 → 收入重要性 → 时点确认
     - 每个关系引用证据卡支撑

数据输出：
  · 完整的标的研究档案（标的 + 证据 + 传导关系）
  → 流向「下一交易日计划」模块
```

#### 阶段二：计划制定（计划模块）

```
数据输入：
  · 持仓数据 (holdings)
  · 候选标的 (plannedAssets)
  · 证据卡 (evidenceRecords)
  · 研究包 (researchSnapshots)
  · 昨日复盘生成的计划草稿 (由盘后模块生成)

数据加工：
  1. 计划编制：计划买入/卖出标的、价格区间、仓位、止损止盈、纪律规则
  2. 版本管理：planVersions（每次修改生成新版本，不覆盖历史）
  3. 计划确认：planConfirmations（确认后 status = active，才可作为执行依据）
  4. 风控规则嵌入：单票仓位、总仓位、单日亏损上限、持股数量上限

数据输出：
  · active 状态的 plans（当前生效计划）
  · planVersions（历史版本可追溯）
  · planConfirmations（确认记录）
  → 流向「盘中执行边界」模块
```

#### 阶段三：盘中执行（盘中执行边界模块）

```
数据输入：
  · 生效计划 (plans where status = active)
  · 当前持仓 (holdings)
  · 当日成交 (trades where date = today)

数据加工：
  1. 计划覆盖检查：哪些持仓有计划，哪些没有
  2. 执行边界校验：
     - 是否在计划价格区间内
     - 是否超过计划仓位
     - 是否违反单日亏损限制
     - 是否超过最大持仓数量
  3. 成交录入：手动录入成交 → 自动触发纪律检查
  4. 纪律事件触发：disciplineEvents（偏离/违规记录）

数据输出：
  · trades（成交记录，只追加）
  · disciplineEvents（纪律事件）
  · 触发 discipline engine 生成 disciplineAssessments
  → 流向「盘后核对与复盘」模块
```

#### 阶段四：盘后复盘（盘后模块）

```
数据输入：
  · 当日成交 (trades where date = today)
  · 当日计划 (plans)
  · 纪律评估 (disciplineAssessments)
  · 最新研究 (researchSnapshots)

数据加工：
  1. 成交核对：手续费补录、成交事实确认
  2. 逐笔纪律评估：
     - 调用 discipline-engine
     - 评估维度：计划一致性、风险控制、执行质量、情绪管理等
     - 输出：disciplineAssessments（每笔成交对应一条评估）
  3. 复盘记录写入 dailySessions[date].postmarket：
     - 事实校正 (factsCorrection)
     - 执行证据 (executionEvidence)
     - 偏离原因 (deviationContext)
     - 改进规则 (correctionRule)
  4. 明日计划生成：基于复盘 → 调用 /api/plans/generate-from-review → 生成草稿

数据输出：
  · disciplineAssessments（纪律评估结果）
  · dailySessions（每日复盘记录）
  · plans（明日计划草稿，需人工确认）
  → 反馈回「研究准备」和「计划制定」阶段，形成闭环
```

### 3.3 资讯信息流（独立管道，全程支撑）

```
资讯采集 (information-collector.js)
    │
    ├─ 多适配器支持：RSS/Atom, JSON Feed, NewsNow, SEC EDGAR, Federal Register
    ├─ 随机时间窗口采集（避免被封）
    └─ 去重：原始指纹 (originalHash) + 来源游标
        ▼
资讯正文调度 (information-content-scheduler.js)
    │
    ├─ 按优先级排队抓取正文
    ├─ 完整文章 / 摘要 / 仅标题 三级区分
    └─ 内容去重与质量评估
        ▼
AI 资讯整理 (information-processing.js)
    │
    ├─ 生成输入哈希 (inputHash) + 规则版本 (ruleVersion)
    ├─ 输出结构：
    │   ├─ shortSummary（一句话摘要）
    │   ├─ topicTags（主题标签）
    │   ├─ industryTags（行业标签）
    │   ├─ assetCodes（关联标的）
    │   ├─ holdingRelevance（持仓关联：关系类型、相关度评分）
    │   ├─ attentionPriorityScore（阅读优先级）
    │   └─ direction（方向：正面/负面/中性/待定）
    └─ 处理状态：pending / processing / completed / failed
        ▼
影响评估确认 (informationImpactConfirmations)
    │
    └─ 人工确认 AI 初评结果
        → 确认后写入 influenceAssessments（高级证据）
        → 可被研究包和计划引用
```

    ├─ AI 整理 ───────► information-processing ──► aiEnrichment
    │
    ├─ 研究包导入 ────► research-import ──► plannedAssets + evidence + relations
    │
    ├─ 影响评估 ──────► influence-engine ──► influenceAssessments
    │
    └─ 概率研报 ──────► probability-engine ──► probabilityReports

### 5.2 不可变账本 (Ledger)

```
ledger.baseline（迁移基线）
    ├─ availableCash   起始可用资金
    ├─ realizedPnl     起始已实现盈亏
    └─ holdings[]      起始持仓

ledger.events[]（事件流，只追加不修改）
    ├─ trade                成交事件
    ├─ fee_adjustment       费用调整
    ├─ funding              资金转入/转出
    ├─ position_recon       持仓校正
    └─ cash_recon           现金校正

投影重建 (rebuildLedgerProjection)：
  baseline + events = 当前 holdings + account 状态
  → 确保任何时候都可以从事件流重建当前状态
  → 支持任意时点的历史回溯（未来扩展）
```

## 六、当前架构存在的问题与优化建议

### 6.1 已识别的问题

**问题 1：资讯与标的的行业关联不够顺畅**
- 现状：informationEvents.aiEnrichment.industryTags 和 plannedAssets.industry 都是自由文本字段
- 问题：新增标的时，行业名称可能与 AI 打标签不一致，导致匹配遗漏
- 已优化：前端增加了模糊匹配（互相包含判断），但本质还是文本匹配

**问题 2：数据模型分散，缺乏统一的实体标识体系**
- 现状：标的标识在不同模块中是 code（证券代码），但代码格式不统一
- 问题：6位数字 vs 带市场前缀（SH/SZ），不同模块可能有不同表示
- 风险：跨模块关联时可能出现匹配失败

**问题 3：资讯影响评估的证据链不够闭环**
- 现状：influenceAssessments 是独立记录，与 evidenceRecords 关联不强
- 问题：影响评估结果如何转化为可复用的证据卡，路径不清晰

**问题 4：研究包 → 计划 → 成交 → 复盘 的数据链路有断点**
- 现状：每个模块相对独立，数据是"流入"关系而非"引用"关系
- 问题：一笔成交对应的计划版本、研究依据，追溯起来较困难

**问题 5：单一 store 对象承载所有数据**
- 现状：前端通过 /api/store 一次性加载全部数据
- 问题：数据量增长后（特别是资讯事件），加载速度会变慢
- 风险：未来可能需要分页/增量加载

### 6.2 优化建议（按优先级）

**高优先级：行业标签体系统一**
```
建议引入 industry_tags 字典表：
  · 预定义一级/二级行业分类（参考申万或证监会行业分类）
  · informationEvents.industryTags 从字典中选择
  · plannedAssets.industry 也从字典中选择
  → 从根本上解决"文本模糊匹配"的不可靠问题
```

**中优先级：成交-计划-研究的引用链强化**
```
建议在 trade 记录中增加：
  · planId + planVersion（引用计划版本）
  · researchSnapshotId（引用研究包，可选）
→ 实现从一笔成交向上追溯到计划和研究依据
→ 让复盘更有针对性
```

**中优先级：影响评估 → 证据卡的转化路径**
```
建议增加"将影响评估转为证据卡"的功能：
  · 一条确认后的 influenceAssessment 可转为 evidenceRecords
  · 类型 = analysis，来源 = 资讯影响评估
  · 自动关联 assetCode、industryTags
→ 让资讯影响真正进入研究体系，而不是停留在资讯页
```

**低优先级：数据加载策略优化**
```
当前数据量不大时可保持现状
未来资讯事件超过 5000 条后考虑：
  · 资讯列表分页加载
  · 详情页按需加载正文
  · 仅加载最近 N 天的资讯
```

## 八、2026-07-23 架构 V2 实施结果（本节优先于前文旧建议）

前文第六章记录的是改造前诊断。以下五项已经落地，后续代理不得再按“待实现”理解。

### 8.1 精简存储

- `app_state` 只保存小型核心状态；
- 资讯事件、采集/处理批次、影响确认、事件簇、证据转化、概率报告、结算记录和审计日志写入 `state_collection_records`，每条记录独立更新；
- 内部全量状态快照改为每 25 次普通修订一次，重要迁移、恢复、导入和校正仍强制留档；
- 自动 JSON 恢复点最多每 6 小时一份，危险操作前强制生成；
- `POST /api/storage/compact` 会先备份，再清理重复副本并执行 SQLite 压缩；
- `GET /api/storage/status` 用于检查数据库、可回收空间、状态版本和备份占用。

真实迁移完成后，数据库由约 361 MB 降至约 30 MB，备份由 60 份降至 14 份。

### 8.2 统一资产与决策主线

- 新增 `assets[]` 资产主档；
- A 股资产 ID 统一为 `CN:{SH|SZ|BJ}:{6位代码}`；
- holdings、plannedAssets、trades、evidenceRecords、companyRelations、计划规则、资讯持仓影响、概率研报均保留兼容代码，同时写入 `assetId`；
- 每笔成交包含 `decisionContext`，冻结计划版本/哈希、研究版本、证据、已确认资讯影响、成交前持仓和账户摘要；
- 旧成交以 `captureMode=legacy-backfill` 补齐可确认的引用，不伪造无法还原的历史盘中事实。

正确追溯路径：

```text
assetId
  → researchSnapshot / evidenceRecords
  → planVersion
  → trade.decisionContext
  → disciplineAssessment
  → postmarket review
```

### 8.3 资讯事件聚类与证据转化

- 不删除不同来源的原始资讯；
- 相同或高度相似报道进入 `informationEventClusters`，概率引擎按事件簇防止重复计权；
- 用户确认 AI 持仓影响后，系统自动创建：
  1. 一张来源可追溯的事实证据卡；
  2. 一张引用事实卡的持仓影响分析卡；
  3. 一条 `informationEvidencePromotions` 转化记录；
- 用户拒绝影响判断时不生成证据；
- 高级 `influenceAssessments` 仍保留，作为需要完整公司传导链时的专家评估，不再是普通用户必填项。

### 8.4 实验贝叶斯更新

- `probabilityReports` 同时保存先验、逐证据似然比、每一步后验和最终实验概率；
- 启发式似然比限制在保守范围，不能由单条消息推到极端概率；
- 相同 `eventClusterId/correlationGroup` 的重复报道只取信息量最强的一条；
- 未通过校准前必须显示“实验贝叶斯”，不得显示为正式概率；
- 影响分、纪律分和阅读优先级都不能直接当作概率。

### 8.5 结算与自动校准

- 实验概率也允许到期结算，以积累冷启动样本；
- 结算保存冻结概率、实际情景、Brier 分和概率来源；
- `GET /api/probability-calibration?horizon=5d` 返回同周期样本统计；
- 至少 30 个已结算样本、平均 Brier 分优于基线、期望校准误差不超过 0.15，才允许升级为 `validated`；
- 校准数据由系统计算，用户不再手填模型编号、样本数或 Brier 分。

### 8.6 当前仍需继续演进的项目

- `/api/store` 仍作为兼容接口返回完整聚合视图；后续页面应逐步切换为按模块读取和游标分页；
- 行业分类字典仍未标准化，但其优先级低于资产身份、证据血缘和事件聚类；
- 自动行情结算可在免费行情来源稳定后增加；当前结算仍要求核对真实收益和基准收益。
